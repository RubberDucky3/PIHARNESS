Overview

PIHARNESS is a bash-based worker orchestration harness that runs alongside your Claude Code session in cmux. It manages Pi worker panes so the orchestrator never writes code directly — every task is delegated to an isolated worker pane.

Plan
Break work into parallel slices

Analyze the task, split into independent units, decide worker count and runtime per unit.

Execute
Spawn workers, send tasks

Spawn workers in parallel (max 3), send prompts, wait for completion, collect results.

Learn
Auto-detect patterns, extract skills

Track tasks, detect repetitive patterns, extract reusable skills, grow the system.

Workflow

1Plan 2Spawn 3Task 4Wait 5Collect 6Commit 
# 1. Check current state
$ ./piharness.sh status

# 2. Spawn workers in parallel (max 3)
$ ./piharness.sh spawn --label impl-structure --worktree
$ ./piharness.sh spawn --label impl-logic --worktree

# 3. Send tasks
$ ./piharness.sh task surface:5 "Build the HTML structure"
$ ./piharness.sh task surface:6 "Implement the API logic"

# 4. Wait for completion
$ ./piharness.sh wait surface:5 --timeout 300
$ ./piharness.sh wait surface:6 --timeout 300

# 5. Collect and compare
$ ./piharness.sh collect surface:5
$ ./piharness.sh collect surface:6
$ ./piharness.sh compare surface:5 surface:6

# 6. Apply best result and commit
# (orchestrator reviews output, applies to repo)

Role-Based Pipeline

For complex features, spawn workers with specific roles and run a full implement → test → review cycle. The pipeline automates the round-trip across all three roles.

# Spawn role-specific workers with isolated worktrees
$ ./piharness.sh spawn --role implementer --worktree --label impl-1
$ ./piharness.sh spawn --role tester      --worktree --label test-1
$ ./piharness.sh spawn --role reviewer    --worktree --label review-1

# Run automated pipeline
$ ./piharness.sh pipeline "Deploy Express API" --rounds 3

# Or orchestrate manually
$ ./piharness.sh orchestrate surface:7 "Review and merge changes"

Command Reference

Spawn / Register

spawn [--cwd DIR] [--label L] [--worktree] [--branch B] [--role ROLE]

Create a new Pi worker pane. First worker splits right, subsequent ones split down. Use --worktree for isolated git branches.

use <surface> [--label L]

Register an existing cmux pane as a named worker.

Task Lifecycle

task <surface> <prompt>

Send a fire-and-forget task to a Pi worker.

auto <surface> <prompt>

Task with automatic runtime failover on error. Recommended for production use.

wait <surface> [--timeout N]

Block until the worker task completes (default 300s).

collect <surface>

Get the complete output from a finished task.

peek <surface>

Read partial output while the worker is still running.

Skill Management Self-Evolving

skill list

List installed skills with version, usage count, and description.

skill show <name>

Show skill details. Increments usage counter.

skill new <name> [--pattern P] [--desc D]

Create a new skill scaffold with SKILL.md and template.md.

skill extract <surface> [--name N]

Create a skill from a completed worker's output.

skill suggest

Get skill suggestions from task history patterns.

learn track <desc> [--surface S] [--outcome O]

Log a task for pattern detection.

learn history [--limit N]

View recent task history.

learn suggest

Analyze history for repeated patterns and suggest new skills.

Observability & Comparison

status

Dashboard: surface, label, status, elapsed, role, runtime, last output line.

screen <surface>

Live terminal capture of the worker pane.

list

Full table of registered workers with metadata.

compare <s1> <s2>

Side-by-side output comparison.

diff <s1> <s2>

Git diff between worktree branches.

close <surface> [--keep-worktree]

Exit a worker pane and clean up. Use --keep-worktree to preserve the branch.

clean

Reset all state: outputs, logs, and worker registry.

Architecture

The orchestrator reads CLAUDE.md, consults the skill registry, and routes tasks to Pi workers in cmux panes. Skills are stored in ~/.piharness/skills/ and grow over time through pattern extraction. 

┌──────────────────────────────────────────────────────┐
│               ORCHESTRATOR (Claude Code)              │
│  Reads CLAUDE.md, consults skill registry,            │
│  routes tasks to workers                              │
└──────────────────────┬───────────────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────────────┐
│              SKILL-AWARE ROUTER                       │
│  task → match against skills → matched? → use skill  │
│  task → no match → new pattern → create in worktree  │
└──────────────────────┬───────────────────────────────┘
                       │
          ┌────────────┼────────────┐
          ▼            ▼            ▼
   ┌──────────┐ ┌──────────┐ ┌──────────┐
   │ Skill #1 │ │ Skill #2 │ │ Skill #N │  ← learned over time
   └──────────┘ └──────────┘ └──────────┘
          │            │            │
          └────────────┼────────────┘
                       ▼
┌──────────────────────────────────────────────────────┐
│              GIT WORKTREE ENGINE                      │

Runtime Chain

When a model hits rate limits or errors, the runtime chain automatically fails over to the next available tier:

pi:kilo-auto/free
▼ failover
pi:stepfun/step-3.7-flash:free
▼ failover
opencode:claude-haiku-4-5
▼ failover
ollama:qwen2.5-coder:7b
▼ failover
claude:default

Override with PIHARNESS_RUNTIMES env var. Use ./piharness.sh runtimes to inspect.

Configuration

PIHARNESS is configured through environment variables. All paths default to ~/.piharness/ unless overridden.

Variable Default Description
PIHARNESS_DIR~/.piharnessState directory for registry, outputs, logs

PIHARNESS_RUNTIMESpi:...→claudeRuntime failover chain (colon-separated)
PIHARNESS_MAX_WORKERS3Maximum concurrent worker panes
PIHARNESS_ORCHESTRATORchain[0]Runtime to use for orchestrator role
